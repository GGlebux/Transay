# Transay — контекст для Claude

Веб-сервис расшифровки медицинских анализов. Монорепозиторий: бэкенд + фронтенд + Docker Compose.
Этот файл — рабочие заметки, **коммитится в репозиторий** (вынут из `.gitignore`). Язык общения с пользователем — **русский**.

## Стек и структура
- `backend/` — Spring Boot 3.4.7 · Java 23 · PostgreSQL 16 · Flyway · JWT. Пакет `project.assay`.
- `frontend/` — React 19 · Vite 7 · TypeScript (отдаётся nginx). Строгий TS:
  `noUnusedLocals`, `noUnusedParameters`, `verbatimModuleSyntax` (типы импортировать через `import type`).
  Сборка: `tsc -b && vite build` — неиспользуемые импорты/локали **ломают билд**.
- Прод: `docker-compose.yml` (готовые образы DockerHub). Dev-сборка: `docker-compose.dev.yml` (тегает `:latest`).

## Локальная среда (важно!)
Локально **нет Node, нет Java/Maven** — только Docker. Проверять сборку через контейнеры:
```bash
# фронт (tsc + vite). node_modules уже стоит в frontend/ после первого прогона
docker run --rm -v "C:\Users\kapit\Documents\Transay\frontend:/app" -w /app node:20 npm run build
# первый раз: ... node:20 bash -lc "npm install --no-audit --no-fund && npm run build"

# бэк (компиляция), m2 кэшируется в volume transay_m2
docker run --rm -v "C:\Users\kapit\Documents\Transay\backend:/app" -v "transay_m2:/root/.m2" -w /app \
  maven:3.9-eclipse-temurin-23 mvn -q -B -o -DskipTests clean compile
```
Артефакты `backend/target`, `frontend/dist`, `frontend/node_modules` — в `.gitignore`, в коммит не идут.

## Деплой / CI
- `git push` в `main` → CI собирает и пушит образы `gglebux/transay-backend|frontend:latest` в DockerHub.
- На сервере: `docker compose up -d --pull always`. Автообновление — профиль `autoupdate` (watchtower).
- Локальный прод-стек: собрать через dev-compose (тегает `:latest`), затем `docker compose up -d` БЕЗ `--pull`.
- Вкладка «Мониторинг» в админке: Actuator + http_log + Portainer (`:9443`).

## Доменная модель (ключевое)
- `Customer` 1—1 «своя» анкета через `customer.person_id` (OneToOne). Это профиль в `/cabinet`, анализы в `/measures`.
- **Семья**: добавлен `person.owner_id → customer` (ManyToOne, миграция `V11`). Члены семьи = люди с `owner_id`,
  кроме своей анкеты. Проверка владения — `PeopleService.getOwnedPerson(personId)`.
- `Person.condition` (enum `Condition`: BASE/GRAVID/MENSES). **Сейчас всегда BASE** — GRAVID/MENSES это задел
  на будущее, в формах НЕ выставляем (см. ниже про баг подбора индикаторов).
- Подбор индикатора: `IndicatorRepository.findAllCorrect` фильтрует по `rusName`, `units`, `gender`(или BOTH),
  **`condition` (точное совпадение)**, возраст BETWEEN minAge..maxAge. Если condition ≠ BASE, а индикаторы
  в БД BASE → «индикатор не найден».
- Сводная таблица / расшифровка — SQL-функции `get_summary_table(person_id)` (V8), `get_decrypt(person_id, date)` (V7).
  Берут только `person_id`, не зависят от customer → для членов семьи работают как есть.

## Дорожная карта (обсуждено 2026-06, перед паузой)
Здесь зафиксированы задумки и принятые решения по 6 крупным фичам — чтобы мысли не потерялись.
**Два ключевых инсайта:**
- **Задачи 3 и 6 — это ОДНА система** (настраиваемый движок правил над значениями индикаторов). Текущая
  логика `Transcript (индикатор + направление FALL/RAISE) → Reason` — это уже примитивный rule-engine на
  один индикатор; косвенные причины, формулы и предупреждения = его обобщение.
- **Задача 4 (синонимы) — фундамент для задачи 5 (парсинг фото):** распознанные имена сопоставляются с
  каноническими индикаторами через таблицу алиасов.

**Рекомендуемый порядок реализации** (от простого/изолированного к сложному):
1. №4 синонимы (быстро, фундамент) + №2 rate-limit 429 (быстро, изолированно).
2. №1 диапазон дат в decrypt (изолированно, понятная семантика).
3. №3+№6 движок правил (модель → SQL-функция → форма-конструктор → блок на дашборде). Ядро ценности.
4. №5 парсинг фото (опирается на №4 и таблицу единиц; самый рискованный — в конце).

### 1. Диапазон дат в Decrypt
Сейчас `get_decrypt(person_id, date)` (V7) берёт ровно одну дату — но за один поход сдают не все показатели.
- **Семантика:** «за период берём ПОСЛЕДНЕЕ значение каждого индикатора в диапазоне»
  (`DISTINCT ON (indicator_id) ... ORDER BY indicator_id, reg_date DESC`). Даёт целостную актуальную картину.
- **API/UI:** `get_decrypt(person_id, date_from, date_to)`; пресеты 3 / 6 / 12 мес / «всё время» + кастомный
  from–to. Фронт — в `.decrypt-controls` (`styles/person.css`).
- **Срок годности — мягкая пометка, НЕ жёсткий фильтр.** Добавить `indicator.relevance_days` (общий дефолт
  ~180, админ переопределяет: ОАК — недели, гликир. гемоглобин — 90, липиды — месяцы). Значение старше срока —
  показываем, но с пометкой «данные могли устареть».

### 2. HTTP 429 (rate limiting)
- В Spring встроенного нет. **Bucket4j + Caffeine (in-memory)** — лёгкий, под одну ноду. При масштабировании
  на несколько инстансов Bucket4j переключается на Redis без переписывания логики.
- **Ключи:** email-действия (регистрация, сброс пароля, повторная отправка кода) — по `email + IP`;
  общий API — по IP или user-id.
- Ответ `429 Too Many Requests` + заголовок `Retry-After`; добавить ветку в `GlobalExceptionHandler`.
- Кандидаты под лимит в первую очередь: `AuthController` (register/login), повторная отправка кода
  подтверждения, forgot-password (всё, что дёргает `EmailSender`).

### 3 + 6. Движок правил: косвенные причины, формулы, предупреждения (ОДНА система)
Единая модель **Rule** (обобщает Transcript→Reason):
- **Условие** — набор предикатов `{индикатор, оператор, порог}`. Операторы: `<`, `>`, `between`,
  отклонение в % (относительно реф-диапазона), статус `FALL/RAISE/OK`. Порог абсолютный (`>200`, `<0.3`,
  `<25%`) или относительный. Предикаты соединяются `AND`/`OR`.
- **Результат (outcome)** — тип:
  - `CAUSE` — конкретная косвенная причина → в доп. список причин рядом с прямыми;
  - `WARNING` — предупреждение → в блок «Предупреждения» на дашборде;
  - `RECOMMENDATION` — «проверить индикаторы X, Y, Z» (как стрелки «проверить» на схемах).
- **Формула** — частный случай: вычисляемый индикатор (выражение над другими, напр. `Общий холестерин /
  ЛПВП`); его результат тоже гоняется через правила (`>3` → WARNING «группа риска»).
- **Цепочки/графы со скринов** («эритроциты ↑ → болезни почек → проверить креатинин») = правило с
  предикатами + RECOMMENDATION. Многоступенчатость («…и если креатинин тоже отклонён») = правило с двумя
  предикатами через AND.

**Хранение/вычисление — гибрид SQL + JSON-конфиг** (как и текущая миграция логики в SQL):
- Модель данных проектируем **граф-готовой** (узлы-условия / узлы-исходы / рёбра), чтобы потом одну
  структуру использовать и как логику, и как картинку пользователю. Черновик таблиц:
  `rule(id, name, outcome_type, outcome_text, priority, enabled)`,
  `rule_predicate(id, rule_id, indicator_ref, operator, threshold, threshold2, logic, grp)`,
  `rule_recommendation(rule_id, indicator_ref)`, `computed_indicator(id, name, expression)`.
  *(Открытый вопрос: нормализованные таблицы vs одно JSONB-определение на правило. Для формы-конструктора
  нормализованные проще; для графа гибче JSONB. Решить при реализации.)*
- **Вычисление — SQL-функция** (по образцу `get_decrypt`/`get_summary_table`): на входе измерения человека,
  прогоняет правила, отдаёт JSON. Полноценный Java rule-engine (Drools) — оверкилл, НЕ нужен.
- **Конструктор на админке — РЕШЕНО: форма-конструктор сначала** («ЕСЛИ индикатор [оператор] [порог] И … ТО
  [результат] + проверить …»). Быстрее и надёжнее; визуальный граф-редактор (React Flow) с переиспользованием
  как картинки для пользователя — следующий этап, модель данных к нему уже готова.
- **Отображение пользователю — РЕШЕНО: блок на сайте/дашборде** (без email/push пока). CAUSE → доп. список
  причин, WARNING → блок «Предупреждения», RECOMMENDATION → «рекомендуем проверить …».

### 4. Синонимы названий индикаторов
Разные больницы/врачи называют один показатель по-разному (проблема всплыла при ручном вводе).
- **Таблица алиасов**, не «унификация» (бланки больниц мы не контролируем):
  `indicator_alias(id, indicator_id, alias_name, source)` где `source = MANUAL | PARSED`.
- **Поиск:** точное совпадение + нечёткое через `pg_trgm` (триграммы Postgres); показывать группу рядом с
  названием («Гемоглобин — Общий анализ крови»), чтобы пользователь не путался.
- **Наполнение:** вручную админом; позже — авто-подтягивание из распознанных бланков (самообучение словаря).
- Это фундамент для №5.

### 5. Парсинг фото лабораторных бланков (самое сложное)
**Пайплайн (от выбора движка распознавания НЕ зависит):**
1. фото → экстрактор → черновик `[{имя, значение, единица, дата}]`;
2. маппинг имён через алиасы (№4) + fuzzy;
3. нормализация единиц — таблица конверсий per-indicator `unit_conversion` (коэффициент/формула, напр. глюкоза
   mg/dL↔mmol/L зависит от вещества); незнакомая единица → спросить пользователя;
4. **ОБЯЗАТЕЛЬНЫЙ экран подтверждения (human-in-the-loop — это мед-данные!):** «распознали X → сопоставили с
   индикатором Y, значение Z ед.», неуверенные сопоставления подсвечиваем, пользователь правит;
5. сохранение в сводную таблицу (создание `measure`/`referent` на дату).

**Движок распознавания — ВЫБОР ОТКРЫТ (решить при реализации):**
- **Claude vision** — фото → структурированный JSON сразу, лучшая устойчивость к грязным фото/рукописи/разным
  формам/русскому; нужен API-ключ и бюджет на фото.
- **Yandex Vision OCR** — хорошо читает русский, но разбор таблиц/структуры — на нашей стороне.
- (Tesseract локально — дёшево, но хрупко; как запасной.)
Точность реализации этого узла отложена — пайплайн и подтверждение от него не зависят.
